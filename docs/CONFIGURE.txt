Configuration Reference (.env)
==============================

The gateway reads settings from ../syno-mihomo-gateway-data/.env. The proxy
subscription URL is stored at
../syno-mihomo-gateway-data/config/subscription.txt. This sibling directory is
not replaced when you update the release folder.

You normally set these through the installer prompts. This file documents what
each value means so you can review or hand-edit .env if needed. After editing
.env by hand, apply changes with "4) Modify an existing deployment" so the
containers are redeployed with the health-gated rollback.

The file is parsed as dotenv data, not executed as shell code. Keep one
KEY=VALUE per line. Installer-written quoted values preserve spaces, &, #, $,
quotes, and backslashes in passwords and secrets.

--------------------------------------------------------------------------
Network settings
--------------------------------------------------------------------------
  ROUTER_IP
      Your router / gateway IP address. The installer auto-fills this from the
      chosen interface during deploy; you confirm it.

  SUBNET_CIDR
      Your home LAN subnet in CIDR form, e.g. 192.168.1.0/24. Auto-filled from
      the chosen interface during deploy; you confirm it.

  MIHOMO_IP
      A FREE static LAN IP for the gateway. It must be on your LAN subnet and
      not used by any other device. Clients point their gateway and DNS here
      to be routed. During deploy the installer warns if this IP is already
      answering on the LAN.

  PARENT_INTERFACE
      The macvlan parent NIC the installer saved from the interface scan.
      Blank = auto-detect.

  TPROXY_DRIVER
      L2 driver for the gateway network: macvlan (default) or ipvlan. Keep
      macvlan -- it is the only driver that routes LAN clients, and a
      macvlan child is reachable from the LAN even on an Open vSwitch
      parent. ipvlan is a dashboard-only escape hatch the installer offers
      on some OVS setups; it does NOT forward LAN-client traffic through
      the proxy.

--------------------------------------------------------------------------
Mihomo TUN (transparent gateway)
--------------------------------------------------------------------------
  TUN_ENABLE
      Transparent-gateway TUN. ON by default (true). This IS the transparent
      gateway: the rendered config carries a tun: block on the "system" stack
      (with allow-lan: true and fake-ip DNS), so LAN devices set their gateway
      and DNS to MIHOMO_IP and route to the internet through the airport. They
      can also use MIHOMO_IP:7890 (http) / :7891 (socks) as an explicit proxy.
      The "system" stack does NOT hijack the controller's reply path, so the
      dashboard at MIHOMO_IP:CONTROLLER_PORT stays reachable -- this is how
      mihomo #1493 is actually fixed, so keep TUN ON. Setting TUN_ENABLE=false
      runs a plain, non-gateway proxy (reachable only via the
      redir/tproxy/mixed/socks ports, no transparent interception of LAN
      clients). Only the lowercase values true and false are accepted.

  TUN_AUTO_REDIRECT
      Only used when TUN_ENABLE=true: optional Linux TCP redirect optimization.
      Default false for DSM compatibility. Set true only when the installer's
      disposable iptables compatibility probe succeeds. Only true/false.

--------------------------------------------------------------------------
Installer
--------------------------------------------------------------------------
  INSTALLER_LANG
      The installer UI language: en or zh. Saved from the first-run language
      prompt so later runs skip it.

--------------------------------------------------------------------------
Dashboard / controller
--------------------------------------------------------------------------
  WEB_UI_PORT
      The dashboard port published on the NAS. Default 8080. Open the
      dashboard at http://<NAS-IP>:<WEB_UI_PORT>.

  CONTROLLER_PORT
      The proxy controller port. Default 9090. Used when adding a backend in
      the dashboard.

  CONTROLLER_SECRET
      Optional dashboard auth secret; empty = no dashboard auth. The
      installer offers to auto-generate a random secret when left empty
      (stored here); the post-deploy summary says where it lives. On a
      re-run with a saved secret, pressing Enter at the prompt keeps it and
      typing '-' clears it. If set, enter it as the backend Secret in the
      dashboard. It must NOT contain a '|' character.

--------------------------------------------------------------------------
DNS settings
--------------------------------------------------------------------------
  DNS_DEFAULT_NAMESERVER
  DNS_NAMESERVER
  DNS_CN_NAMESERVER
  DNS_FOREIGN_NAMESERVER
      Comma-separated lists of DNS servers. These control how the gateway
      resolves names; a bad DNS configuration can leave the gateway
      unhealthy after deploy. The shipped defaults are mainland-China-safe
      and privacy-hardened (split-horizon v2): every entry is an IP or an
      encrypted DoH URL on an IP (nothing needs resolving first), and an
      entry ending in #All Nodes is dialed THROUGH the proxy tunnel - via
      the hidden All Nodes group, a full-pool auto-picker kept solely as
      this DNS anchor (the dashboard does not show it, and DNS keeps
      working whatever you pick in the dashboard). DNS values from an
      existing install need no change for the streamlined group model.
      DNS_CN_NAMESERVER / DNS_FOREIGN_NAMESERVER are the split-horizon
      pair: Chinese-list names resolve only domestically, and EVERYTHING
      ELSE - foreign-list names AND names on no list at all - resolves
      only via the tunnel; if the airport is completely down, those
      lookups fail instead of leaking to a domestic resolver. Both are
      REQUIRED - split-horizon v2 is the only DNS profile, and the
      renderer refuses to render without them.
      Keep DNS_NAMESERVER domestic and never #-suffixed: it also resolves
      the airport node names BEFORE any proxy exists, so a tunneled entry
      there can dead-end a fresh start.

  Who can see your DNS queries (with the shipped defaults)
      * AliDNS (Alibaba) / DNSPod (Tencent): ONLY names on the China list,
        over encrypted DoH (plus the few bootstrap hosts the gateway
        itself needs). Nothing else reaches them - a dnsleaktest.com
        extended test from a LAN device shows no domestic resolver.
      * Your ISP / anyone on the wire: only that the gateway talks to
        well-known resolver IPs over encryption - not the names you
        look up.
      * The airport operator: the foreign traffic it proxies anyway,
        which now includes your tunneled DNS.
      * Cloudflare / Google (the foreign resolvers): foreign and unlisted
        names, arriving through the tunnel - they see the airport's
        address, not your home address.
      The doctor reports which profile is live as "dns_privacy".

  DNS_GEOIP_NO_RESOLVE
      true stops the routing rule from ever resolving unlisted names,
      even through the tunnel (zero local resolution). With the v2
      defaults that lookup is already private, so keep false - a Chinese
      site missing from the built-in China list then keeps loading
      directly instead of riding the proxy (see TROUBLESHOOTING.txt).
      Only true or false.

  SNIFFER_ENABLE
      true (the default) turns on traffic sniffing: connections that
      arrive as bare IPs - from devices that resolve DNS outside the
      gateway - get their real hostname read from the connection
      itself, so the routing rules still apply to them (streaming
      reaches its group, poisoned lookups get corrected at the node).
      This repairs ROUTING for such devices; their DNS privacy still
      requires pointing the device's DNS at the gateway (see
      TROUBLESHOOTING.txt, "LAN clients bypass the gateway's DNS").
      false renders without the sniffer (pre-1.3.10 behavior).

  FULL_PROXY_SOURCES
      Optional per-device full-proxy band: comma-separated IPv4
      addresses/CIDRs (a bare IP means /32; e.g. 192.168.1.240/28).
      Renders only when set -- unset keeps the rendered config
      byte-identical (the feature is invisible). When set, a
      "Full Proxy" selector appears on the dashboard and one
      SRC-IP-CIDR rule per entry lands immediately after the LAN
      rule: band devices reach LAN destinations DIRECT, and
      everything else -- streaming and CN sites alike -- rides
      Full Proxy (strict semantics). Malformed values refuse to
      render naming this key -- IPv6 entries, hostnames, octets
      >255, a prefix >32, duplicates, whitespace, backticks (IPv6
      is refused on purpose: routable LAN IPv6 routes around the
      IPv4-only gateway -- see caveat 3 in the "Full-proxy
      devices" section below). Per-device mode switching is
      router-side (a DHCP fixed-IP flip), never a gateway restart.
      See the "Full-proxy devices (FULL_PROXY_SOURCES)" section
      below.

  COUNTRY_GROUPS
      REQUIRED - the dashboard's group model is built from it (an
      empty value refuses to render; start from the shipped default):
      NAME=regex;NAME=regex creates one "<Country> Auto" auto-
      selecting group per entry - the panel shows NAME and keeps it
      on the fastest node whose name matches regex - and those groups
      are the members of the "Country Pick" selector, where you
      choose the exit country. The FIRST entry is the out-of-box
      default country (the shipped default puts Japan Auto first). A
      regex naming several countries IS a multi-country group. The
      shipped example fits airports that name nodes in Chinese
      (美国01, 新加坡05...) - edit the entries to match YOUR
      airport's naming. A group that matches no node blocks traffic
      when selected instead of silently bypassing the proxy (the
      doctor flags it; when the empty group is the country
      Country Pick is riding, default-route traffic is blocked - pick
      another country there while you fix the pattern). Names may
      contain spaces ("Japan Auto") but must not start/end with one,
      nor reuse a built-in, reserved, or retired group name
      (All Nodes / Country Pick / Proxy Mode / Streaming Sites /
      DIRECT / REJECT / ... - the render error names the collision;
      the full list is in .env.example); backticks and malformed
      entries refuse to render.

  Proxy groups on the dashboard
      Proxy Mode is the mode selector the routing rules target:
      default Country Pick, plus DIRECT / REJECT / the raw airport
      nodes (pin ONE exact node here when a site objects even to
      same-country IP changes). Streaming Sites is the per-service
      selector for unlock-sensitive sites (Netflix + the audio
      services; default Proxy Mode). Country Pick chooses the exit
      country: its members are the "<Country> Auto" groups above, so
      general traffic keeps a single exit IP and the exit country
      never hops on its own. All Nodes still exists but is hidden -
      it only anchors the DNS detour above. Naming legend:
      "<X> Auto" = auto-pick, "<X> Mode" = mode selector,
      "<X> Sites" = site rules, "<X> Pick" = manual choice.

  Removed keys
      The old priority include/exclude filter pair (and the older
      AUTO_EXCLUDE_FILTER) are gone - the "<Country> Auto" groups
      plus Country Pick replace them. A .env still carrying any of
      those lines refuses to render; the error names the exact lines
      to delete (the gateway keeps running on its previous config
      meanwhile). After this upgrade, dashboard selections reset
      ONCE (saved picks are keyed by group name) - re-pin them in
      the dashboard.

      These settings are for the GATEWAY only. The NAS's OWN DNS lives in
      DSM Control Panel > Network > General and must also point at
      resolvers reachable from your network - a NAS whose only resolver is
      unreachable cannot reach its vendor services or update anything
      (see TROUBLESHOOTING.txt; the doctor probes the NAS resolvers too).

General

  TZ
      Timezone for the containers, e.g. Asia/Shanghai.

--------------------------------------------------------------------------
Image source
--------------------------------------------------------------------------
  REGISTRY_MODE
      Where images are pulled from:
        acr    - (DEFAULT) pulls from YOUR private image mirror. Recommended
                 in mainland China where the big public registries are
                 blocked.
        docker - upstream public registries. Only works on a NAS with
                 unfiltered internet.

  MIHOMO_TAG / METACUBEXD_TAG / CF_TAG
      Image tags for each component. Default "latest".

  DOCKER_REGISTRY
      Your registry host (used in acr mode).

  DOCKER_USERNAME
      Your registry username (used in acr mode).

  ACR_PASSWORD
      Your registry password (used in acr mode).

  ACR_NAMESPACE
      Your registry namespace (used in acr mode).

  In acr mode you provide host + namespace + username + password once and the
  installer derives the full image references for you.

--------------------------------------------------------------------------
Automatic updates
--------------------------------------------------------------------------
  UPDATE_ENABLED
      Turns scheduled auto-updates on or off.

  UPDATE_IMAGES
      Space-separated image references the update job checks and pulls.
      Leave it inheriting the three image variables (the default
      "${MIHOMO_IMAGE} ${METACUBEXD_IMAGE} ${CF_IMAGE}"): the updater
      refuses to run if this list stops including MIHOMO_IMAGE and
      METACUBEXD_IMAGE exactly (and CF_IMAGE, when that is set).

  UPDATE_DENY_CONTAINERS
      Optional space-separated glob denylist for generic auto-update
      targets: an enrolled container whose name matches any pattern is
      never auto-updated. Example: "critical-* my-database". See
      AUTO-UPDATE.txt.

  UPDATE_SCHEDULE
      A validated five-field numeric cron expression. Default is 02:00 daily.
      Configure the DSM task to match it.

  UPDATE_TZ
      Timezone used for updater log timestamps. DSM fires the task in the NAS
      Regional Options timezone. Default Asia/Shanghai.

  DOCKER_READY_TIMEOUT / DOCKER_READY_INTERVAL
      Seconds a scheduled or boot task waits for Container Manager, and the
      retry interval. Defaults: 120 / 5.

  EXPECTED_ARCH
      The expected CPU architecture: amd64 or arm64. Set this to match your
      NAS to catch architecture mismatches early.

  NOTIFY_WEBHOOK_URL
      Optional webhook URL (Bark / Gotify / Slack-compatible JSON POST) for
      update reports. When set it fires on failures AND successful updates,
      independent of DSM's own notifications -- it is the reliable channel.
      Leave empty to rely on the DSM task's "send run details by email".

  NOTIFY_ON_NOCHANGE
      Set to 1 to also get a notification when an update run changed
      nothing. Default 0 (quiet on no-op runs).

  UPDATE_LOG / LOG_KEEP
      The updater's rotating log file and how many rotations to keep
      (default 7). A relative UPDATE_LOG lands under the data directory:
      the default ./logs/auto-update.log means
      ../syno-mihomo-gateway-data/logs/auto-update.log.

--------------------------------------------------------------------------
Optional / advanced: tunnel
--------------------------------------------------------------------------
  CF_* settings
      A tunnel component is optional and advanced. Leave the CF_* settings at
      their defaults unless you specifically need a tunnel. During deploy the
      image step asks "Also manage a cloudflared tunnel container? (advanced,
      optional)" -- default No. Answering yes collects CF_TAG and a tunnel
      token (Enter reuses the token of an already-running container).

  CF_DNS
      Optional: comma-separated IPv4 resolvers passed to the tunnel
      container as docker --dns, so it stops inheriting the NAS's own
      resolv.conf (an unreachable NAS resolver otherwise silently kills the
      tunnel). Takes effect on the next provision or blue-green update
      (sudo sh scripts/gateway.sh update --yes). Empty keeps the current
      behavior. Suggestion: mirror DNS_NAMESERVER.

--------------------------------------------------------------------------
Full-proxy devices (FULL_PROXY_SOURCES)
--------------------------------------------------------------------------
  Optional. Setting FULL_PROXY_SOURCES reserves a small IPv4 band whose
  devices skip the smart routing above: one SRC-IP-CIDR rule per entry is
  spliced immediately after the LAN rule, so a band device still reaches
  LAN destinations DIRECT, but everything else -- streaming and CN sites
  alike -- rides the "Full Proxy" group (strict semantics: no CN
  short-circuit for these sources). Unset, nothing renders and the config
  stays byte-identical -- the feature is invisible until you opt in.

  The "Full Proxy" group is a dashboard selector. Members:
      * Proxy Mode (the default) -- the band follows whatever the
        dashboard's Proxy Mode says.
      * every "<Country> Auto" group -- pin the band's exit country
        independently of the main Country Pick choice.
      * REJECT -- the kill switch: one click takes the whole band
        offline.
  There is deliberately no DIRECT member: a band device can never be
  silently un-proxied by a dashboard click -- leaving the band is a
  router-side act (below), never a selector state.

  Prerequisite: DHCP fixed-IP reservations are the router's job. The
  gateway does not control leases -- it only matches source IPs. Reserve
  a small fixed-IP band on your router (e.g. 192.168.1.240/28) and set
  FULL_PROXY_SOURCES to it.

  Switching a device in or out is router-side -- verified on UniFi
  (per-client Fixed IP reservation), and any router with per-client DHCP
  reservations works the same way:
      1. Flip the device's fixed-IP reservation INTO the band and
         reconnect the device (toggle Wi-Fi / re-plug) so it picks up
         the new lease: the device is now full-proxy.
      2. Flip the reservation back OUT of the band and reconnect: back
         to normal rule-based routing.
  No gateway or container restart is ever involved -- mode switching is
  entirely a router-side lease flip.

  Changing the band itself (the .env value) is a normal .env edit; apply
  it with "4) Modify an existing deployment"
  (sudo sh scripts/gateway.sh redeploy --yes). The render gate applies:
  a malformed value refuses to render, naming FULL_PROXY_SOURCES, and
  keeps the previous config running. Rejected loudly: IPv6 entries,
  hostnames, octets >255, a prefix >32, duplicates, whitespace,
  backticks. IPv6 entries are refused on purpose -- matching is
  IPv4-only, and routable LAN IPv6 routes around the gateway entirely
  (caveat 3 below).

  The doctor gained a "full_proxy" check: disabled (knob unused) / ok /
  parity-drift (the rendered rules no longer match the knob -- your band
  edit is NOT live, redeploy) / chain-violation (a non-LAN flow from a
  band source bypassed Full Proxy).

  Caveats:
      1. CN sites/apps on band devices exit overseas. Under fake-ip the
         node resolves band devices' domains remotely -- CN domains
         correctly ride the tunnel; expect CN services to see an
         overseas exit and degrade accordingly. That is the strict
         semantics working as designed, not a leak.
      2. QUIC/UDP fallthrough. A UDP flow whose exit node lacks UDP
         relay falls through the SRC-IP-CIDR rule at the rule engine --
         CN-listed destinations then go DIRECT; browsers retry over
         proxied TCP. The doctor's full_proxy check flags such flows;
         an opt-in UDP block knob ships only if an airport's UDP proves
         unreliable.
      3. The guarantee assumes no routable LAN IPv6 (doctor
         ipv6_bypass) -- a dual-stack device with global IPv6 routes
         around the IPv4-only gateway entirely.

--------------------------------------------------------------------------
Subscription URL
--------------------------------------------------------------------------
  The proxy subscription URL is NOT in .env. It lives in:
        ../syno-mihomo-gateway-data/config/subscription.txt
  The installer seeds this file and asks for the URL during deploy. A bad or
  unreachable subscription URL is a common reason the gateway is unhealthy
  after deploy.
