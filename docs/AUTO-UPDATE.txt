Automatic Updates
=================

The gateway can keep its images up to date on a schedule. An updater script
pulls the latest images, redeploys ONLY what changed, runs a health check, and
automatically ROLLS BACK if the new image turns out to be unhealthy.

You set this up from the installer:

      sudo sh ./install.sh
      -> 3) Set up automatic updates (cron)

--------------------------------------------------------------------------
Choosing the time and timezone
--------------------------------------------------------------------------
The cron setup asks for the daily run time as HH:MM (24h). Pressing Enter
accepts the default shown in the prompt (your current schedule; 02:00 on a
fresh install). Times with a leading zero, such as 08:30, are stored
correctly. Next comes the timezone menu: Asia/Shanghai is listed first,
followed by other common zones and an "Other (type it)" entry. This menu has
NO Enter-default - you must type the number of your choice. The timezone
only labels the updater's log timestamps (see below).

The installer then asks "Enable automatic updates?" (default Yes; answering
No writes UPDATE_ENABLED=false, so the scheduled job will no-op until
re-enabled) and, optionally, "review which OTHER containers auto-update
(generic targets)?" (default No; answering Yes opens the generic-target
management flow - see "Generic targets" below).

--------------------------------------------------------------------------
Recommended: DSM Task Scheduler (survives system upgrades)
--------------------------------------------------------------------------
Note: do NOT use Container Manager's "Project" Build/Update as an update
method - it has no digest gate, no health check, and no rollback. The
scheduled script below is the safe path, and its non-zero exit codes drive
DSM's "send run details" email (the default way you get notified).

On Synology the recommended, upgrade-persistent way to schedule updates is a
user-defined script in DSM:

      Control Panel -> Task Scheduler -> Create -> Scheduled Task
                    -> User-defined script

Run it as root. When you choose the DSM web UI method, the installer prints
the EXACT settings to enter (the command line, schedule, and user). Copy those
into the Task Scheduler task.

The printed command uses an absolute working directory and an explicit /bin/sh,
as recommended for DSM 7 tasks. Do NOT add a ">> ...log" redirection of your
own: the updater already writes and rotates its log at
../syno-mihomo-gateway-data/logs/auto-update.log. A second redirection
duplicates log lines and keeps writing to the old file after rotation.

Enable "Send run details only when the script terminates abnormally", or use
Task Scheduler Settings -> Save output results. After creating the task, select
it and click Run once; verify a zero result and check
../syno-mihomo-gateway-data/logs/auto-update.log.

Why the GUI task is preferred: a crontab line may be overwritten by DSM during
a system upgrade. The Task Scheduler task persists across upgrades.

--------------------------------------------------------------------------
How to schedule (an explicit choice)
--------------------------------------------------------------------------
After the questions above, the installer asks "How do you want to schedule
it?". The menu repeats until you choose Done, so you can combine actions
(for example run the dry-run first, then set up the DSM task):

  1) Set it up via the DSM web UI (Task Scheduler) - recommended
     Upgrade-persistent. The installer prints the exact Control Panel ->
     Task Scheduler settings to enter.
  2) Install a crontab entry now (CLI; DSM may overwrite it on upgrade)
  3) Validate now with a dry-run (pulls cache; does not swap containers)
     Use this before trusting the schedule.
  4) Done

Choosing Done is checked against what actually happened: if updates are
enabled but no crontab entry was installed and the DSM instructions were
never shown, the installer warns that nothing will actually run and offers
to set UPDATE_ENABLED=false first. If the instructions were shown, it
reminds you that the task must still be created in the web UI or updates
will not run.

--------------------------------------------------------------------------
IMPORTANT: timezone vs actual trigger time
--------------------------------------------------------------------------
The timezone you chose labels the in-job log timestamps. The actual trigger
time follows the NAS SYSTEM timezone: DSM Task Scheduler uses Regional Options,
and BusyBox cron also uses the system timezone. Set the NAS time correctly.

--------------------------------------------------------------------------
Related .env settings
--------------------------------------------------------------------------
  UPDATE_ENABLED    - turns scheduled updates on/off.
  UPDATE_SCHEDULE   - a cron expression for when updates run (default 02:00
                      daily).
  UPDATE_TZ         - timezone used for updater log timestamps (default
                      Asia/Shanghai); DSM scheduling uses the NAS timezone.
  EXPECTED_ARCH     - amd64 or arm64; set to match your NAS.
  TUN_AUTO_REDIRECT - optional TCP redirect optimization; default false on
                      DSM. true runs an iptables compatibility probe first.
  NOTIFY_WEBHOOK_URL - optional webhook (Bark/Gotify/Slack-compatible JSON
                      POST), the reliable push channel for run reports; DSM's
                      own push is best-effort only.
  NOTIFY_ON_NOCHANGE - set to 1 to also be notified when nothing changed
                      (default 0).

See CONFIGURE.txt for the full settings reference.

--------------------------------------------------------------------------
What the updater does on each run
--------------------------------------------------------------------------
  1. Validates all settings, waits for Container Manager, validates Compose,
     and logs in to ACR before pulling.
  2. Pulls each image with retries, then verifies its local image ID and
     architecture.
  3. Compares against what is running and redeploys only changed components.
     Compose v2 uses "up -d --pull never" so the checked local image cannot be
     replaced by a second implicit pull.
  4. If TUN_AUTO_REDIRECT=true, proves the target image can create a NAT chain
     against the DSM kernel in a disposable network namespace before apply.
  5. Checks the controller, TUN dataplane, forwarding, and crash-loop state.
  6. If Compose fails or a new image is unhealthy, verifies/re-tags the previous
     image, recreates without pulling, and runs the health check again.

For external cloudflared, the updater starts a conflict-free temporary connector
without the canonical static IP or published host ports and proves it connected.
It then rebuilds and verifies the canonical container with the latest image and
the saved ports/networks. If finalization fails, it restores the previous image
and run specification while the temporary connector keeps the tunnel live. A
candidate that may be the only connector is never removed automatically.
If the update cannot complete, the run summary distinguishes three outcomes:
ROLLED BACK to the previous connector (now connected); FAILED with the
candidate discarded (the old connector untouched); or FAILED needing manual
attention (the -candidate container may be the only live connector - do not
remove it).

A dry-run DOES update the local image cache, but never swaps a container.

--------------------------------------------------------------------------
Updater exit codes
--------------------------------------------------------------------------
  0  Success or no-op (nothing needed updating).
  2  Partial failure (some components updated, others did not).
  3  Config / preflight error (check .env and prerequisites).
  4  Another run is already in progress.
  5  Registry login failed (check your registry credentials).

If the schedule reports a non-zero code, see TROUBLESHOOTING.txt and check
../syno-mihomo-gateway-data/logs/auto-update.log.

--------------------------------------------------------------------------
Logs
--------------------------------------------------------------------------
The updater writes its log to ../syno-mihomo-gateway-data/logs/auto-update.log
(the persistent data folder next to the release folder - nothing is logged
inside the release folder). Review it after a scheduled run to confirm what
changed and whether a rollback occurred.

Each run ends with a summary: counts (updated/unchanged/failed/rolled_back)
followed by one line per target. The same outcome, including the names of
updated, failed, and rolled-back targets, is saved to
../syno-mihomo-gateway-data/state/last-run.json; view it with:

  sh scripts/gateway.sh update --last

--------------------------------------------------------------------------
Generic targets (any enrolled container)
--------------------------------------------------------------------------
Besides the gateway containers, the updater can refresh ANY running container
whose image is already on your ACR. Enroll/manage with:

  sudo sh scripts/gateway.sh update --enable <container> --yes
  sh scripts/gateway.sh update --list-targets
  sh scripts/gateway.sh update --last
  sudo sh scripts/gateway.sh update --disable <container> --yes

or via the installer's update flow. Containers that are compose-managed,
ambiguous, matching UPDATE_DENY_CONTAINERS, or carrying settings the engine
cannot replay are refused with a reason. Database-like images warn loudly:
recreation has no quiesce. Updates run serially (generic targets, then
cloudflared, then the gateway pair LAST), each behind a health gate with
automatic restore of the saved container on failure. In the run summary, a
target the engine refused to touch is reported as REFUSED (container
untouched - check the log, or de-enroll it), distinct from FAILED/ROLLED
BACK. See CLI.txt for the full gateway.sh reference.

--------------------------------------------------------------------------
Optional: canary-verify an update (advanced)
--------------------------------------------------------------------------
Not required for normal use. If you want proof that an update preserved a
container's configuration, snapshot it before and compare after:

1. Dry-run smoke:   sudo sh scripts/auto_update.sh --dry-run
2. Canary:          sudo sh scripts/state_diff.sh snapshot <name> /tmp/before
                    sudo sh scripts/gateway.sh update --yes
                    sudo sh scripts/state_diff.sh compare <name> /tmp/before
   Any "DRIFT:" line means a setting was not preserved (narrow exception:
   labels/healthcheck artifacts are image-relative overrides and can drift
   falsely when the new image bakes in the same value - verify with
   docker inspect).
3. Gateway checks:  run sudo sh scripts/gateway.sh status and
                    sudo sh scripts/gateway.sh doctor (full reference in
                    CLI.txt); if anything looks wrong, see
                    TROUBLESHOOTING.txt.
