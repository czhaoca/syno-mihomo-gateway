故障排查
========

下面每一条的格式为：现象 -> 可能原因 -> 解决办法。日志写入
../syno-mihomo-gateway-data/logs/（本文件夹的同级目录）：安装程序写入
install.log，计划更新任务写入 auto-update.log，gateway.sh 写入
gateway.log（若 gateway.sh 先运行过，它会把另外两个名字链接到 gateway.log，
三者共享同一文件）。出问题时请先查看与你所用工具同名的日志。

--------------------------------------------------------------------------
安装程序拒绝运行 / 要求你移动文件夹
--------------------------------------------------------------------------
  原因：文件夹不在 Docker 共享文件夹之下。Docker 守护进程会从
        docker-compose.yml 所在位置绑定挂载文件，因此它必须位于 docker
        文件夹之下（通常为 /volume1/docker；卷号可能不同）。
  解决：移动它，例如   sudo mv "<here>" "/volume1/docker/"   然后从新位置
        重新运行 install.sh。你也可以在 File Station 中把它拖入 "docker"
        共享文件夹。

--------------------------------------------------------------------------
"权限被拒绝" / 需要特权的步骤失败
--------------------------------------------------------------------------
  原因：未以 root 身份运行。部署需要 root 来创建 /dev/net/tun、创建 macvlan
        网络并运行 docker。
  解决：改用   sudo sh ./install.sh

--------------------------------------------------------------------------
docker 或 docker compose 提示 "command not found"
--------------------------------------------------------------------------
  原因：NAS 上未安装或未启用 Container Manager / Docker 套件。
  解决：在 DSM 套件中心安装并启动容器套件，然后重新运行 install.sh。

--------------------------------------------------------------------------
/dev/net/tun 缺失
--------------------------------------------------------------------------
  原因：TUN 设备尚未创建，而网关需要它。它（以及 macvlan 网络）在 NAS
        重启后不会保留。
  解决：以 root（sudo）身份重新运行部署。部署流程会创建 /dev/net/tun。如果
        仍失败，请确认你是 root。
        要在每次重启后自动修复，请按 scripts/install_scheduler.sh 打印的说明
        添加 DSM "开机"（Boot-up）触发的计划任务（用户 = root），它运行
        scripts/setup_network.sh 重建 macvlan 网络和 /dev/net/tun。参见
        AUTO-UPDATE.zh.txt。

--------------------------------------------------------------------------
重启后 TUN 静默失效（doctor 报 "TUN gateway is not ready"）
--------------------------------------------------------------------------
  症状：sudo sh scripts/doctor.sh 报告
        "ERROR in-container TUN gateway is not ready"，其余检查全部通过；
        docker logs mihomo 出现 "Start TUN listening error: configure tun
        interface: no such file or directory"。使用显式代理端口的设备仍然
        正常，因此很容易被忽视。
  原因：Docker 在开机任务创建 /dev/net/tun 之前就自启了 mihomo，容器捕获
        到的是空的设备绑定。事后补建宿主机节点无法修复已在运行的容器。
  解决：sudo docker restart mihomo
        然后验证：docker exec mihomo ls /sys/class/net/ 必须能看到
        mihomo-tun，且 doctor.sh 报告 HEALTHY。预防：把运行
        scripts/setup_network.sh 的开机任务排在 DSM 启动顺序的最前面。

--------------------------------------------------------------------------
mihomo 的 IP 已被占用 / 部署因 IP 冲突失败
--------------------------------------------------------------------------
  现象：安装程序警告你为 mihomo 选择的 IP 已在局域网上应答，或部署因该 IP
        被占用而失败。
  原因：另一台设备（通常由 DHCP 分配地址）已占用 MIHOMO_IP。
  解决：使用菜单选项 2 "使用已保存的 .env 部署（可编辑设置或按原样部署）"。
        它会复用 .env 中保存的一切——路由器、DNS、端口、订阅、镜像设置——
        因此你无需重新输入。在它的五个选项中选择 "更改 mihomo 的 LAN IP，
        然后部署" 并选一个空闲 IP。这是修复 IP 冲突的推荐方式。

--------------------------------------------------------------------------
无法退出安装程序 / 卡在某个提问处
--------------------------------------------------------------------------
  现象：你想退出，但停在某个提问或菜单处。
  解决：在任意提问或菜单处按 Ctrl-D 即可干净地退出安装程序。你也可以选择
        菜单中的 "退出" 项。

--------------------------------------------------------------------------
macvlan 父接口 / 接口不匹配
--------------------------------------------------------------------------
  现象：网络创建失败，或客户端无法访问网关 IP。
  原因：macvlan 网络绑定到了错误的 NAS 接口。
  解决：重新运行部署并重新选择局域网接口，或使用菜单选项 2（使用已保存的
        .env 部署）-> "重新选择网络接口，然后部署"。安装程序会列出每个接口
        及其 IPv4，并标记推荐项。

--------------------------------------------------------------------------
安装程序警告所选接口是 Open vSwitch 端口
--------------------------------------------------------------------------
  现象：选择局域网接口时（以及创建网络前再次），安装程序警告父接口
        （如 ovs_eth0）是 Open vSwitch 端口，并询问是否改用 ipvlan 驱动。
  含义：这是提醒，不是错误。macvlan 在多数 OVS 环境下可正常工作；只有某些
        OVS 配置会让局域网其他设备无法访问 macvlan 子接口（网关/仪表盘从
        客户端超时）。
  解决：先回答 No（默认值），保持 macvlan。只有当部署后局域网设备确实无法
        访问网关时，才重新部署并回答 Yes：ipvlan 可穿越 OVS，但仅限仪表盘
        —— 它不会把局域网客户端流量经代理转发，透明网关功能将失效。

--------------------------------------------------------------------------
无法从 NAS 打开面板
--------------------------------------------------------------------------
  原因：主机隔离。使用 macvlan 时，NAS 主机无法访问网关自己的局域网 IP
        （MIHOMO_IP）。
  解决：通过 http://<NAS-IP>:<WEB_UI_PORT> 打开面板（该地址在 NAS 上可访问）。
        若要测试代理或网关 IP，请使用局域网中的另一台设备。

--------------------------------------------------------------------------
面板“无法连接后端”（从局域网设备访问超时）
--------------------------------------------------------------------------
  原因：TUN 块用的是带 auto-route 的 gvisor/mixed 栈，它可能截走控制器的回包，
        使从局域网设备访问 MIHOMO_IP:PORT 超时（尽管容器内 127.0.0.1 可正常
        响应）。这不是 URL/密钥错误，不是 CORS 拦截，也不是 Open vSwitch 问题。
  解决：保持 TUN 开启（TUN_ENABLE=true，默认值）并使用 "system" TUN 栈。与
        mixed/gvisor + auto-route 不同，system 栈不会截走控制器回包，因此仪表盘
        后端保持可达，同时透明网关转发照常工作。这是经过验证、可正常工作的配置
        —— 不要为绕开此问题而关闭 TUN。重新部署即可。请从另一台局域网设备验证：
        curl http://MIHOMO_IP:CONTROLLER_PORT/version 应返回 JSON。
        （设置 TUN_ENABLE=false 会运行普通的非网关代理，不会透明拦截局域网客户端。）

--------------------------------------------------------------------------
镜像拉取失败
--------------------------------------------------------------------------
  原因 A：acr 模式下仓库凭据错误 -> 更新/登录可能报告退出码 5（仓库登录
          失败）。
  解决 A：重新核对 DOCKER_REGISTRY、DOCKER_USERNAME、ACR_PASSWORD 和
          ACR_NAMESPACE（选项 4 -> 镜像来源），然后重试。
  原因 B：在被封锁的网络环境下（如中国大陆）使用 "docker" 模式，公共仓库
          无法访问。
  解决 B：切换到 "acr" 模式，从你自己的私有镜像源拉取。

--------------------------------------------------------------------------
架构不匹配
--------------------------------------------------------------------------
  现象：容器无法启动，或运行了错误的二进制文件。
  原因：镜像架构与 NAS 的 CPU 不匹配。
  解决：把 .env 中的 EXPECTED_ARCH 设为与你的 NAS 一致的 amd64 或 arm64，
        然后重新部署（选项 4 -> 应用更改）。

--------------------------------------------------------------------------
面板端口已被占用
--------------------------------------------------------------------------
  现象：因端口被占用，容器无法启动。
  原因：另一个服务已占用 WEB_UI_PORT（默认 8080）。
  解决：把 .env 中的 WEB_UI_PORT 改为一个空闲端口（选项 4 -> 应用更改），
        然后打开 http://<NAS-IP>:<新端口>。

--------------------------------------------------------------------------
部署后网关 "不健康"
--------------------------------------------------------------------------
  原因：通常是订阅地址错误/无法访问、DNS 设置有误，或 TUN 故障。
  解决：检查 ../syno-mihomo-gateway-data/config/subscription.txt（订阅地址）以及
        DNS_DEFAULT_NAMESERVER / DNS_NAMESERVER / DNS_FALLBACK 各项，然后
        重新部署。部署与自动更新流程都会对新镜像做健康检查，若持续不健康会
        自动回滚。
        如果日志显示 "iptables (nf_tables): Could not fetch rule set
        generation id"（仅当 TUN_ENABLE=true 时相关），请在 .env 中设置
        TUN_AUTO_REDIRECT=false 后重新部署；system TUN 栈仍会提供网关数据通路。

--------------------------------------------------------------------------
容器看似健康，但局域网客户端无法上网
--------------------------------------------------------------------------
  解决：运行   sudo sh scripts/doctor.sh --egress
        它不仅检查控制器，还会检查 /dev/net/tun 和容器内的 mihomo-tun 数据通路。
        退出码 0 表示健康，2 表示代理出口降级，3 表示本地配置或运行时故障。
        请从另一台局域网设备测试，因为 NAS 无法访问自己的 macvlan 子容器。

--------------------------------------------------------------------------
自动更新报告了非零退出码
--------------------------------------------------------------------------
  0  成功 / 无操作。
  2  部分失败 - 查看 ../syno-mihomo-gateway-data/logs/auto-update.log。
  3  配置 / 预检错误 - 检查 .env 与先决条件。
  4  已有一次运行正在进行中 - 稍后再查看。若上次运行崩溃留下了陈旧的锁，
     下次运行会自动检测并回收它，无需任何手动清理。
  5  仓库登录失败 - 重新核对你的仓库凭据。

--------------------------------------------------------------------------
DSM 7 计划更新没有运行
--------------------------------------------------------------------------
  * 确认任务计划条目已启用，并以 root 运行。
  * 重新运行 scripts/install_scheduler.sh，复制它打印的精确命令。
  * DSM 按 NAS“区域选项”的时区触发；UPDATE_TZ 只改变日志时间戳。
  * 先手动点击一次“运行”，再检查任务结果和
    ../syno-mihomo-gateway-data/logs/auto-update.log。
  * 不要在任务命令后自行添加 ">> auto-update.log" 重定向；这会重复输出并
    干扰更新器自己的日志轮转。

--------------------------------------------------------------------------
应该去哪里查看
--------------------------------------------------------------------------
  * 日志：         ../syno-mihomo-gateway-data/logs/ —— install.log（安装程序）、
                   auto-update.log（计划更新）、gateway.log（gateway.sh；若
                   gateway.sh 先运行过，这些名字会链接到同一个共享文件）。
  * 设置：         ../syno-mihomo-gateway-data（见 CONFIGURE.zh.txt）。
  * 修改并重新部署：install.sh -> 4) 修改已有部署。
